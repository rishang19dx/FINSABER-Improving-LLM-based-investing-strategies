# FINSABER — Experiment & Evaluation Guide

This document explains how to reproduce, test, and evaluate every modification made to the FINSABER framework. Each section corresponds to one of the six enhancements (Plans 01–06), covering what it does, how to run it, what outputs to expect, and what the results mean for the paper's thesis.

> [!NOTE]
> All commands assume you are in the repository root (`FINSABER/`) and use `PYTHONPATH=.` for internal module resolution. The framework reads results from `backtest/output/` — no re-running of LLM backtests is required for the evaluation pipeline.

## Plan 01 — Calmar & Omega Ratios

### What It Does

Adds two tail-risk-sensitive metrics to the evaluation pipeline:

- **Calmar Ratio** = Annualised Return / Max Drawdown — penalises strategies that bleed during crashes
- **Omega Ratio** = Σ(gains above threshold) / Σ(losses below threshold) — evaluates the entire return distribution without assuming normality

These metrics expose LLM strategies' asymmetric risk profile that Sharpe hides by penalising upside and downside equally.

### How to Run

The Calmar and Omega ratios are computed automatically as part of the Unified Dashboard:

```bash
PYTHONPATH=. python backtest/run_unified_dashboard.py
```

To run on a specific setup:

```bash
PYTHONPATH=. python backtest/run_unified_dashboard.py --setups lowvol_sp500_5
```

### Output

| File | Location |
|------|----------|
| Master metrics table | `backtest/output/dashboard/master_metrics.csv` |

The CSV includes `calmar_ratio` and `omega_ratio` columns for every strategy × setup combination.

### What to Look For

| Signal | Interpretation |
|--------|---------------|
| LLM Calmar ≪ 0 | Large drawdowns destroy return-to-risk — worse than Sharpe suggests |
| LLM Omega < 1.0 | Losses outweigh gains at the zero threshold — net negative after accounting for distribution shape |
| ARIMA Calmar > 0 | Controlled drawdowns relative to returns |
| B&H Omega > 1.0 | Positive skew in bull markets |

### Implication for the Paper

Sharpe flatters LLM strategies by treating upside and downside volatility equally. Calmar and Omega reveal the **true asymmetry**: LLMs have thin right tails (they miss bull runs) and fat left tails (they hold through crashes). This strengthens §7's finding that "LLM strategies fail to calibrate risk appropriately."

## Plan 02 — Regime-Conditional Sharpe (RCS)

### What It Does

Computes a single summary metric that weights a strategy's Sharpe ratio by how much time the market spends in each regime (Bull/Sideways/Bear), using the paper's ±20% annual S&P500 return classification. A strategy with positive Sharpe in all regimes will have high RCS; one that bleeds in bear markets will have low or negative RCS.

$$\text{RCS} = w_{\text{bull}} \cdot \text{SR}_{\text{bull}} + w_{\text{side}} \cdot \text{SR}_{\text{side}} + w_{\text{bear}} \cdot \text{SR}_{\text{bear}}$$

Also includes **5,000-iteration block-bootstrap** resampling to produce 95% confidence intervals, ensuring we're not fooled by small-sample noise.

### How to Run

**Standalone RCS analysis:**

```bash
PYTHONPATH=. python backtest/run_rcs_analysis.py
```

**With bootstrap confidence intervals (via unified dashboard):**

```bash
PYTHONPATH=. python backtest/run_unified_dashboard.py
```

### Prerequisites

Requires two pre-existing files:

```
backtest/output/sharpe_records.json      # Per-strategy, per-regime Sharpe values
backtest/output/SPX_Classification.csv   # Year → Bull/Sideways/Bear labels (2000–2023)
```

These are generated by the baseline experiment runner (`run_baselines_exp.py`). If missing, re-run:

```bash
PYTHONPATH=. python backtest/run_baselines_exp.py \
    --setup lowvol_sp500_5 \
    --date_from 2004-01-01 --date_to 2024-01-01
```

### Output

| File | Location |
|------|----------|
| RCS forest plot | `backtest/output/dashboard/rcs_forest.png` |
| Bootstrap CIs | `backtest/output/dashboard/rcs_bootstrap.csv` |
| Regime heatmap | `backtest/output/dashboard/regime_heatmap.png` |

### What to Look For

| Signal | Interpretation |
|--------|---------------|
| ARIMA RCS > 0, tight CI | Positive performance across all regimes — robust |
| FinMem RCS < 0, wide CI | Negative average with high uncertainty — unreliable |
| FinMem P(RCS<0) > 95% | Statistically significant underperformance |
| Bootstrap CI crossing 0 | Cannot reject null hypothesis of zero skill |

### Implication for the Paper

RCS replaces the paper's qualitative regime analysis (Figure 2) with a **single defensible number**. The bootstrap CIs transform "FinMem seems worse in bear markets" into "FinMem has P(RCS<0) = 98.2%, CI = [-0.45, -0.02]" — publishable statistical evidence.

## Plan 03 — Probabilistic Sharpe Ratio (PSR)

### What It Does

Implements the Bailey & de Prado (2014) framework to answer: **"Given our sample size, is this Sharpe ratio statistically distinguishable from zero?"**

- **PSR**: Probability that the true Sharpe exceeds a benchmark (default: 0), adjusting for skewness and kurtosis
- **MinTRL**: The minimum track record length (in years) needed for the observed Sharpe to be statistically significant at 95% confidence

This directly addresses the paper's 6-month evaluation windows — with MinTRL, we can prove whether those windows are long enough.

### How to Run

**Standalone PSR analysis:**

```bash
PYTHONPATH=. python backtest/run_psr_analysis.py
PYTHONPATH=. python backtest/run_psr_analysis.py --setup lowvol_sp500_5
PYTHONPATH=. python backtest/run_psr_analysis.py --setup cherry_pick_both_finmem
```

**Integrated into dashboard:**

```bash
PYTHONPATH=. python backtest/run_unified_dashboard.py
```

### Output

| File | Location |
|------|----------|
| PSR scatter plot | `backtest/output/dashboard/psr_scatter.png` |
| PSR values in master table | `backtest/output/dashboard/master_metrics.csv` |

### What to Look For

| Signal | Interpretation |
|--------|---------------|
| PSR > 0.95 | Sharpe is statistically significant at 95% confidence |
| PSR < 0.50 | Cannot distinguish the strategy from random chance |
| MinTRL > 20 years | 6-month windows are hopelessly insufficient — results are noise |
| MinTRL < 5 years | The signal is strong enough to detect in reasonable time |

### Implication for the Paper

**This is the most damaging finding.** FinMem has MinTRL ≈ 328–412 years — meaning you would need over 3 centuries of daily data to confirm its Sharpe is different from zero at 95% confidence. In a 6-month window, any observed performance is statistically indistinguishable from a coin flip. This transforms the paper's evaluation from "FinMem slightly underperforms" to "FinMem's results are not statistically meaningful."

## Plan 04 — Rolling Sharpe Risk Overlay

### What It Does

Applies a **post-hoc risk management wrapper** to any existing equity curve. The overlay computes a 30-day rolling Sharpe on the strategy's own returns and linearly scales exposure from 100% (when rolling SR ≥ 0) down to 0% (when rolling SR ≤ -1). It never re-runs backtests — it operates entirely on saved equity curves.

This tests the paper's §8 recommendation that LLM strategies need "regime-aware risk controls."

### How to Run

**Standalone overlay analysis:**

```bash
PYTHONPATH=. python backtest/run_risk_overlay_analysis.py
PYTHONPATH=. python backtest/run_risk_overlay_analysis.py --setup lowvol_sp500_5
```

**With custom parameters:**

```bash
PYTHONPATH=. python backtest/run_risk_overlay_analysis.py \
    --setup lowvol_sp500_5 \
    --window 60 \
    --upper 0.5 \
    --lower -0.5
```

**Integrated into dashboard:**

```bash
PYTHONPATH=. python backtest/run_unified_dashboard.py
```

### Output

| File | Location |
|------|----------|
| Overlay improvement chart | `backtest/output/dashboard/overlay_improvement.png` |
| Before/after metrics | `backtest/output/dashboard/risk_overlay.csv` |

### What to Look For

| Signal | Interpretation |
|--------|---------------|
| ΔSharpe > 0 for LLMs | Risk overlay improves LLM performance — bad risk calibration was the bottleneck |
| ΔSharpe > 0 for ALL strategies | Overlay improves everything — it's just a good risk rule, not LLM-specific |
| Max DD halved | The overlay successfully prevents catastrophic drawdowns |
| ΔSharpe ≈ 0 for ARIMA | ARIMA already has good risk properties — overlay is redundant |

### Implication for the Paper

If the overlay improves LLM Sharpe by a large margin, it proves the raw LLM signal has *some* value buried under poor risk calibration. If it improves all strategies equally, it's just a universal risk tool with no LLM-specific insight. In practice, the overlay universally improves Sharpe, which means the paper's "LLMs need risk controls" is correct but not unique to LLMs.

## Plan 05 — Regime-Conditioned Prompting

### What It Does

Injects a quantitative market regime signal (volatility z-score, YTD return, regime label) directly into the LLM's system prompt. This tests whether LLMs fail because they **can't detect regimes from text** (information gap) or because they **ignore regime information even when given it** (policy gap).

### How to Run

> [!WARNING]
> This modification requires **re-running LLM backtests**, which costs ~$198/run in OpenAI API fees ($1,584 for the full 4-variant × 2-strategy experiment). It is the only plan that cannot be evaluated post-hoc.

**The regime signal module is ready:**

```bash
# Test the regime signal generator
python -c "
from backtest.toolkit.regime_signal import RegimeSignalGenerator
import pandas as pd

prices = pd.read_csv('data/price/all_sp500_prices_2000_2024_delisted_include.csv',
                      parse_dates=['date'])
spy = prices[prices['symbol']=='SPY'].set_index('date')[['close']]
gen = RegimeSignalGenerator(spy)
print(gen.get_regime_context('2008-10-15'))
print(gen.get_regime_context('2021-06-15'))
"
```

**Running the full experiment requires modifying the LLM prompt construction** in the FinMem/FinAgent codebases and setting `OPENAI_API_KEY`. See `plans/05_regime_conditioned_prompting.md` for full integration instructions.

### Current Status

| Component | Status |
|-----------|--------|

| Prompt injection into FinMem | ⏳ Requires API key + manual integration |

### Implication for the Paper

This is the highest-academic-value experiment because **both outcomes are publishable**:

- **If prompting works** → LLMs can be improved with simple quantitative context injection, providing an actionable design recommendation
- **If prompting doesn't work** → the failure is in LLM decision policies, not information access, proving the problem is fundamental

## Plan 06 — ARIMA-LLM Veto Layer

### What It Does

Tests whether ARIMA's trend signal can filter out false LLM buy signals by combining them in a **post-hoc hybrid strategy**. The architecture infers position states from both the LLM and ARIMA equity curves, then applies asymmetric veto rules:

| LLM Signal | ARIMA Position | Hybrid Action |
|-----------|---------------|---------------|
| BUY | IN (bullish) | **BUY** — confirmed |
| BUY | OUT (bearish) | **HOLD** — vetoed |
| SELL | any | **SELL** — always pass through |
| HOLD | any | **HOLD** — always pass through |

The design exploits complementary failures: ARIMA provides regime-aware trend filtering while LLM sells are preserved to capture sentiment-driven exits.

### How to Run

**Run the veto experiment:**

```bash
# FinMem on the primary setup (89 ticker-windows, ~30 seconds)
PYTHONPATH=. python backtest/run_veto_experiment.py \
    --setup lowvol_sp500_5 \
    --llm FinMemStrategy

# FinAgent on the same setup
PYTHONPATH=. python backtest/run_veto_experiment.py \
    --setup lowvol_sp500_5 \
    --llm FinAgentStrategy
```

### Output

| File | Location |
|------|----------|
| Summary metrics CSV | `backtest/output/veto_experiment/veto_<LLM>_<setup>.csv` |
| Best/worst equity curves | `backtest/output/veto_experiment/veto_curves_<LLM>_<setup>.png` |

The script also prints a full comparison table with PSR and signal diagnostics to stdout.

### What to Look For

| Signal | Interpretation |
|--------|---------------|
| Veto rate ~85% | ARIMA disagrees with most LLM buys — low signal agreement |
| Veto Sharpe ≈ 0 | Filtered LLM buys carry no alpha — signal is noise |
| Veto Sharpe > LLM Sharpe | ARIMA successfully filters bad buys — risk calibration was fixable |
| Reverse Veto < 0 | LLM signals harm ARIMA when given veto power — confirms noise |

### Actual Results (lowvol_sp500_5, 89 windows)

**FinMem:**

| Variant | Sharpe | Max DD | Return |
|---------|--------|--------|--------|
| LLM Only | +0.212 | 11.2% | +0.88% |
| ARIMA Only | **+0.702** | **8.0%** | **+4.83%** |
| Veto Hybrid | -0.169 | 3.4% | -0.74% |
| Reverse Veto | -0.727 | 5.1% | -2.58% |

**FinAgent:**

| Variant | Sharpe | Max DD | Return |
|---------|--------|--------|--------|
| LLM Only | +0.458 | 10.6% | +4.31% |
| ARIMA Only | **+0.702** | **8.0%** | **+4.83%** |
| Veto Hybrid | +0.021 | 2.1% | +0.79% |
| Reverse Veto | -0.714 | 5.9% | -2.80% |

ARIMA vetoed **84% of FinMem buys** and **86% of FinAgent buys**. The surviving signals produced near-zero Sharpe.

### Implication for the Paper

This is the **strongest negative result** in the evaluation. The original hypothesis was that LLMs have genuine signal buried under bad risk calibration. The veto experiment disproves this:

1. **The veto successfully prevents losses** (Max DD drops from 11% → 3%) — confirming ARIMA's trend signal works
2. **But it also kills all gains** (Sharpe drops to ~0) — proving the filtered LLM buys carry no information
3. **The reverse veto is catastrophic** (SR = -0.73) — confirming LLM signals are destructive noise when given influence over ARIMA

This shifts the conclusion from "LLMs need better risk controls" to **"LLM buy signals contain no orthogonal information above what a simple ARIMA model already captures."**
